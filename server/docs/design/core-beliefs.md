# 설계 철학 및 에이전트 운영 원칙 (core-beliefs.md)

## 이 프로젝트의 작업 방식

이 프로젝트는 **AI 에이전트가 주도적으로 작업**하되,
사람이 설계 결정, 리뷰, 방향 조정을 담당하는 협업 모델로 운영된다.

"코드를 절대 직접 쓰지 않는다"는 제약은 없다.
단, 에이전트가 더 효율적으로 일할 수 있는 환경을 설계하는 것이 우선순위다.

---

## 핵심 원칙

### 1. 레포가 진실이다
> "에이전트 관점에서, 컨텍스트 안에서 접근할 수 없는 정보는 존재하지 않는 것이다."

- 구글 문서, Slack, 머릿속에 있는 결정은 에이전트에게 보이지 않는다
- 모든 설계 결정, 컨벤션, 아키텍처 선택은 레포 안에 기록한다
- 문서가 없으면 에이전트는 추측한다. 추측은 오류다.

### 2. 실패는 "더 열심히"로 해결하지 않는다
에이전트가 같은 실수를 반복하거나 막힐 때, 해결책은 프롬프트를 바꾸는 게 아니다.
"무엇이 빠졌는가?"를 질문한다:
- 빠진 문서가 있는가?
- 빠진 컨벤션 규칙이 있는가?
- 빠진 린터/검사 도구가 있는가?

그 답을 레포에 추가한다. 에이전트가 그 답을 추가한다.

### 3. 의도는 명시적으로, 강제는 기계적으로
- "좋은 코드를 써라" ← 에이전트에게 의미 없음
- "레이어 의존성을 지켜라. `tools/check_layers.py`가 위반을 감지한다" ← 유효함

규칙은 문서에 쓰는 것으로 끝나지 않는다. CI나 린터로 기계적으로 강제한다.

### 4. 컨텍스트는 작게 유지한다
AGENTS.md는 100줄 이내로 유지한다.
모든 것을 한 파일에 넣으면 에이전트의 작업 컨텍스트를 잡아먹는다.
목차 → 세부 문서 구조를 유지한다.

### 5. 작업 계획은 파일로 남긴다
모든 서버 작업을 시작하기 전, `docs/exec-plans/active/`에 관련 실행 계획 파일을 확인하거나 새로 만든다.
exec-plan 초안을 사용자에게 검토받기 전에는 구현 파일을 수정하지 않는다.
작업이 완료되면 `completed/`로 이동한다.
새 에이전트 세션이 시작될 때 active 플랜을 읽으면 컨텍스트를 빠르게 복원할 수 있다.

## 사람이 에이전트에게 맡기기 좋은 작업

- 반복적인 보일러플레이트 구현 (패킷 핸들러, getter/setter 등)
- 컨벤션 기반의 코드 생성 (FlatBuffers 스키마 → C++ 래퍼)
- 테스트 케이스 작성
- 문서와 코드 간의 불일치 탐지
- 레이어 위반 탐지 및 수정

## 사람이 직접 결정해야 할 것

- 새 도메인 추가 여부
- 외부 라이브러리 도입
- 아키텍처 레이어 변경
- 핵심 알고리즘 설계 (히트 판정 방식, 매칭 알고리즘 등)
