# 설계 철학 및 에이전트 운영 원칙 (core-beliefs.md)

## 이 저장소의 작업 방식

이 저장소는 AI 에이전트와 사람이 함께 작업한다.
사람은 방향, 우선순위, 최종 판단을 맡고, 에이전트는 구현, 정리, 검증 자동화를 맡는다.

---

## 핵심 원칙

### 1. 레포가 진실이다

- 레포 밖의 결정은 에이전트에게 없는 정보로 취급한다.
- 설계 결정, 컨벤션, 제약, 워크플로는 레포 안에 남긴다.
- 문서가 없으면 에이전트는 추측하고, 추측은 회귀를 만든다.

### 2. 실패는 더 강한 프롬프트보다 더 나은 구조로 해결한다

에이전트가 반복해서 막히면 먼저 아래를 점검한다.

- 문서가 부족한가
- 컨벤션이 불명확한가
- 자동 검사가 비어 있는가

부족한 것을 레포에 추가한 뒤 다시 진행한다.

### 3. 의도는 문서로, 강제는 도구로 남긴다

- 추상적 구호보다 기계적으로 검사 가능한 규칙을 우선한다.
- 의존관계, 문서 링크, 인코딩 같은 반복 실수는 공용 도구로 막는다.

### 4. 공용 규칙은 공용 계층에 둔다

- 서버와 `Lib/`가 함께 따르는 규칙은 특정 앱 하위 문서에 두지 않는다.
- 공용 컨벤션, 설계 철학, 검사 도구는 루트 계층으로 올린다.
- 앱 문서에는 앱 고유의 제약과 보충 규칙만 남긴다.

### 5. 컨텍스트는 작게 유지한다

- `AGENTS.md`는 진입과 라우팅에 집중한다.
- 상세 절차, 컨벤션, 철학, 레퍼런스는 별도 문서로 분리한다.
- 항상 필요한 문서와 상황별 문서를 구분한다.

### 6. 작업 계획은 파일로 남긴다

- 각 작업 영역은 자신의 active exec-plan 인덱스를 가진다.
- 구현 전에는 관련 plan을 확인하거나 새로 만든다.
- 사용자가 plan을 검토하기 전에는 구현 파일을 수정하지 않는다.

---

## 사람이 맡아야 할 판단

- 새 도메인 도입 여부
- 외부 라이브러리 채택
- 계층 구조 변경
- 제품/서비스 방향 결정

## 에이전트에게 맡기기 좋은 작업

- 반복 구현과 정리
- 테스트 보강
- 문서와 코드의 불일치 탐지
- 규칙 위반 자동 검사
- 문서 구조와 라우팅 정리
