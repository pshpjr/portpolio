# 서버 아키텍처

## 목적

포트폴리오 서버가 어떤 구조로 시작하고, 어떤 방향으로 확장될지를 기획 관점에서 정의한다.

## 핵심 규칙

- 시작 구조는 모듈형 모놀리스다.
- 네트워크 I/O, 글로벌 작업, 던전 인스턴스 처리를 분리한다.
- 각 던전 인스턴스는 독립적인 단일 스레드 루프에서 동작한다.
- 모듈 간 직접 결합보다 메시지 전달 구조를 우선한다.
- DB 접근은 비동기 처리와 메모리 캐싱을 기본으로 한다.
- 이후 마을 서버와 던전 서버의 물리 분리를 고려한다.

## 주요 흐름

### 런타임 구조

1. I/O 스레드풀이 패킷 송수신과 직렬화를 담당한다.
2. 글로벌 워커가 마을, 채팅, 공용 시스템 작업을 처리한다.
3. 던전 인스턴스 스레드가 전투, AI, 기믹 같은 인던 로직을 순차 처리한다.

### 모듈 간 협력

1. 시스템은 필요 작업을 직접 호출보다 메시지 또는 잡 형태로 넘긴다.
2. 디스패처가 대상 유저나 인던의 위치를 추상화한다.
3. 공용 시스템 요청은 잡 큐를 통해 전달한다.

### 데이터 처리

1. 로그인 시 필요한 유저 상태를 메모리에 적재한다.
2. 전투 중에는 메모리 상태를 기준으로 판단한다.
3. 변경분은 비동기 write-back으로 DB에 반영한다.

## 제약과 예외

- 이 문서는 서버 구현 규칙이 아니라 상위 기획 의도와 구조 방향을 다룬다.
- 스레드 수, 큐 구현, 락 전략 같은 세부 구현은 서버 문서에서 확정한다.
- 경매장 같은 공용 시스템은 현재 기획 예시이며 실제 스코프는 조정될 수 있다.
