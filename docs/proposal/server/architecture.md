# 서버 아키텍처

## 1. 개요
본 프로젝트의 서버 아키텍처는 **모듈형 모놀리스(Modular Monolith)** 구조를 채택한다. 단일 프로세스 내에서 시작하되, 서비스 간 결합도를 낮춰 미래의 **분산 서버(마을/인던 분리)** 로의 전환을 용이하게 설계한다.

## 2. 스레드 모델 (Thread Model)
전체 시스템의 부하 분산과 동기화 병목 최소화를 위해 다음 3가지 유형의 스레드풀을 운영한다.

### 2.1. I/O 스레드풀 (Asio)
- **역할**: 네트워크 패킷 수신/송신, Protobuf 직렬화/역직렬화.
- **특이사항**: 순수 I/O에만 집중하며, 비즈니스 로직은 `Worker Thread`로 전달.

### 2.2. 글로벌 워커 스레드 (Global Worker)
- **역할**: 마을 관리, 채팅, 친구 시스템, **경매장(Auction House)** 처리.
- **동기화**: 다수의 스레드가 공유 데이터를 관리하므로, 필요시 `Mutex` 또는 `Lock-free Queue`를 사용.

### 2.3. 인던 인스턴스 스레드 (Dungeon Instance Thread)
- **역할**: 각 던전 인스턴스의 게임 루프(AI, 전투 판정, 기믹) 수행.
- **핵심 정책**: **인스턴스당 싱글 스레드(Single-threaded per Instance)**.
  - 해당 던전에 속한 모든 엔티티(플레이어, 몬스터, 기믹)는 단일 스레드 내에서 순차적으로 처리된다.
  - **이점**: 레이스 컨디션 차단, 나중에 던전 서비스를 별도 프로세스로 떼어내기 매우 유리함.

## 3. 모듈간 통신 (Inter-Module Communication)
나중에 물리적 서버가 분리되더라도 코드 수정이 최소화되도록 **메시지 패싱(Message Passing)** 구조를 지향한다.

- **Dispatcher**: 특정 유저나 던전이 어느 스레드/서버에 있는지 몰라도 메시지를 보낼 수 있는 추상화 레이어 제공.
- **Job Queue**: 글로벌 시스템(경매장 등)에 요청을 보낼 때 직접 호출하지 않고 `Job` 객체를 큐에 삽입.

## 4. 데이터 레이어 및 비동기 처리
- **DB 접근**: 모든 DB I/O는 전용 스레드풀에서 **비동기(Asynchronous)**로 처리된다.
- **메모리 캐싱**:
  - **로그인 시**: 유저의 인벤토리, 장비, 모든 무기별 스킬트리 정보를 메모리에 전량 로드.
  - **인던 내**: DB 접근을 원천 차단하고 메모리 데이터만 사용하여 전투 판정.
  - **로그아웃/주기적 저장**: 변경된 데이터를 DB에 비동기 Write-back.

## 5. 미래 확장 전략 (Post-Portfolio)
1. **Dungeon Server**: 인던 인스턴스 관리 모듈을 독립된 프로세스로 분리.
2. **Gateway Server**: 인증 및 패킷 라우팅 담당 서버 추가.
3. **Redis**: 서버 간 유저 세션 및 랭킹 정보 공유를 위한 캐시 서버 도입.
