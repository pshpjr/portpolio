## 전원 합의 (5/5) — 구현 블로커

### R-1. §2.3 3풀 스레딩은 권장안과 완전히 다른 아키텍처

의견의 "io풀 / 던전 풀 / 던전 밖 세션 풀"은 권장안(IO 1개 + 로직 1개)과 "튜닝 차이"가 아니라 소유권·큐잉·순서 보장·디버깅 방식이 전부 달라지는 수준의 설계 분기다.

빠진 계약:

- 던전 입장/퇴장 시 큐 재바인딩의 원자성 보장 (drain-then-switch 등)
- IO풀→로직풀 핸드오프 큐 구조 (lock-free MPSC / mutex / asio::post)
- 단일 TCP 소켓에 대해 단일 스레드에서만 send 호출한다는 보장
- 각 풀 스레드 수 산정 근거 (2000 접속 기준)
- IO풀에서 DB 처리 여부 — "미정"이 아니라 성능상 분리 필수
- 종료 시 drain 규칙, backpressure 정책

> **보강 요청**: 스레드 도메인별 책임, 세션/던전/DB 작업 소유권, ingress/egress 큐 구조, 순서 보장 범위를 별도 표로 확정

- 던전 입장 퇴장 패킷 수신 시 오류 생길 수 있는 점 인지
  - 던전 추후 scaleout 고려해서 큐 바인딩 구조 생각했었음
- io풀 -> 로직 풀 : session의 connectionID로 moduler 연산 -> 담당 스레드 판별 후 해당 큐에 push (mpsc)
- 단일 TCP 소켓에 멀티스레드가 send 호출 당연히 가능
- 2000 기준 io스레드 hardware concurrency만큼. 로직은 0.5 hardwareconcurrency . 테스트 해 봐야 알지만 IOBound가 큰 스레드는 더 많이 생성
- DB 스레드 풀 생성 시 0.5 hardwareConcurrency로 처리. db callback 처리는 미정
- sendQ backpressure 시 session disconnect 및 패킷 drop

### R-2. §3.1 EntityID 인스턴스 로컬 시 구역 전환 ID 충돌

의견은 "던전 인스턴스 내 고유"이지만, 이 결정의 파급효과가 문서화되지 않았다.

- 구역 전환 시 이전 구역의 잔여 패킷이 새 구역에서 같은 EntityID에 잘못 적용되는 race condition. "미등록 EntityId 무시" 규칙은 새 구역에서 같은 ID가 등록되면 무효
- 파티/채팅/로그 등 서버 전역 참조 시 `(InstanceId, EntityId)` 복합키 필요
- 로그에 unqualified EntityID만 찍히면 디버깅 불가
- EntityID 재사용 정책 미정

> **보강 요청**: `EntityId`를 전역 유지할지 `(InstanceId, EntityId)` 복합키로 갈지 확정. 구역 전환 시 remap 규칙, stale packet drop 기준, 로그/텔레메트리 키도 함께 정의

- EntityID 64 byte 상위 32비트에 객체 타입 등 추가. 애초에 이전 구역은 다른 프로세스가 될 것. 그걸 고려해서 답한 것
- 로그에 타입이나 닉네임 적어야지 누가 EntityID만 적고 있음.
- EntityID 재사용은 안하는데 인던끼린 겹칠 수 있다고

### R-3. §4.3 필드 "확정" 표시인데 의견이 근거를 부정

GuildName, Velocity, CurrentSkillId, StatusEffectFlags가 "확정"으로 표시되어 있으나, 바로 아래 의견이 존재 근거 자체를 부정한다.

- `CurrentSkillId` (int32 하나) → 다단 히트/연계 스킬 단계 표현 불가. 별도 스킬 시전 이벤트 패킷 필요
- `StatusEffectFlags` (uint32 비트) → 최대 32종, 중첩 횟수/잔여 시간 표현 불가. 개별 적용/해제 이벤트 필요
- `Velocity` FVector → 2차원 이동이면 FVector2D. 공중 기믹 유무 먼저 확정
- `GuildName` → 길드 시스템이 v1 범위인지 확인. 아니면 제거
- `MoveTimestamp` → §3.1 의존인데 구조체에 이미 존재
- 스폰 시 1회 전달(DisplayName, GuildName 등)과 매 이동 전송 필드가 구분되지 않음

> **보강 요청**: 필드를 `런타임 필수` / `파생 가능` / `코스메틱` / `v2`로 재분류. 스폰 전용 vs 이동 패킷 필드 분리 명세 추가

스킬 쪽 잡고 있는 중인데 맘대로 추가한 것. 공중 기믹 없고, movetimestamp 제거. statusEffect나 currenctSkillID는 제거

## 4/5 합의 — 주요 문제

### R-4. §1.1 프레이밍 계약의 자기모순

- 권장안: length = 페이로드만, 최대 64KB. 의견: length = 헤더 포함 전체, 최대 16KB
- length가 헤더 포함이면 최소 유효 패킷은 4B. length=0~3은 불법. 유효성 규칙 없음
- 16KB 상한 근거 없음. 일괄 스폰 200엔티티 ≈ 24KB > 16KB. 분할 프로토콜 필요 여부

> **보강 요청**: 헤더 포함/미포함 단일 선택 확정. length 유효 범위, 초과/decode 실패 처리 규칙을 표로 정리. 16KB 상한의 근거(최대 단일 메시지 크기 추정)

- 가장 패킷이 큰 상황은 유저 외형 400b 정도를 받을 때. 주변에 10명이면 4K, 160명 있으면 64K
- 하지만 서버는 최소한의 정보만 주고 디테일 외형 정보는 클라 요청 시 준다고 했음. 160명 한 번에 주면 애초에 클라도 랜더링 한 번에 못함
- 뭐 그래 패킷 헤더 포함 64K로 해줄게

### R-5. 세션 상태 머신 부재

문서 전체에서 세션의 상태 전이가 한 번도 정의되지 않았다. §1.3에서 INVALID_SESSION 에러 코드를 넣었지만 "미인증 상태"의 정의가 없다.

> **보강 요청**: `Connected → Authenticated → InLobby → InGame → Closing` 전이도 + 상태별 허용 패킷 화이트리스트 + 불허 패킷 수신 시 동작(에러 응답 후 유지 vs 즉시 disconnect)

- closing 상태가 필요할진 모르겠고, 등록 안 된 패킷 쏘면 에러 응답

---

### R-6. §3.2 외형 정보 요청의 N+1 문제

"id 배열을 보내면 각각 응답" 방식의 위험:

- 50명 구역에 1명 진입 시 49건 요청은 감당 가능
- 50명 동시 전환 시 50×49 = 2450건 동시 요청 폭탄
- 응답 전 Actor의 기본 외형(빈 상태)이 정의되지 않음
- 응답 전 엔티티가 디스폰되면 응답 무시 규칙 필요
- 외형 변경(장비 교체) 시 push인지 재요청인지 미정

> **보강 요청**: v1은 "스폰 패킷에 최소 외형 포함" 또는 "배치 응답 1회"로 단일화. 개별 요청이면 배치 상한·rate limit 규칙 추가

- 기본 외형은 종족, 이름 정도만.
- 외형 받을 액터 배열로 정보 받는다고 했던 것 같고. 그러면 요청 50개 + nfy 50번이 다인 것 같은데
- 외형 변경 시 클라가 알아서 표시할 수 있지 그걸 서버에 왜 물어봄. 그냥 유저 정보를 lod처럼 단계 나눠서 요청하라고 클라에서.

### R-7. §1.3 에러 전달 경로 3종의 역할 불명

현재 에러 전달 경로가 3개:

1. 각 RES 패킷 내 에러 코드 enum
2. ErrorResponse 공통 패킷 ("특이 상황"용)
3. Disconnect 사유 Enum (§1.1)

"특이 상황"의 정의가 없고, Disconnect 사유를 클라이언트에 전달하는 패킷 형태도 미정. DDoS 같은 서버 내부 사유를 클라이언트에 노출하면 보안 문제.

> **보강 요청**: 에러 경로별 사용 조건 확정. 클라이언트 노출용/서버 내부 로그용 Enum 분리 여부 결정

- disconnect 사유를 왜 클라에 보내냐. 어떻게 보내려고
- 특이 상황은 구현하며 추가될 예정. 컨텐츠와 관련되지 않은 에러 처리

---

### R-8. 구역 전환 시 대량 스폰 vs 16KB 제한 충돌

200 엔티티 스폰 ≈ 24KB > 16KB 제한. 분할 전송 시 ZoneTransitionBegin/End 사이에 수십 패킷이 끼어들고, 클라이언트 상태 머신(Transitioning → WaitingForNewZone → InZone) 전이 규칙이 없음.

> **보강 요청**: 전환 중 클라이언트 상태 전이도, 타임아웃 정책, 보존/초기화 상태 경계(파티·채팅 보존, 엔티티 초기화) 정의

- 전환 요청 시 전환중이라는 플래그는 하나 있어야 할 듯. 전환 타임아웃 30초로 걸고

## 3/5 합의 — 보통 문제

### R-9. §2.2 목표 규모 10명 vs 2000명

권장안 10명과 의견 2000명은 200배 차이. 세션 정책이 아니라 부하 모델 자체가 미정.

- 2000명 한다고

### R-10. Ping/Pong 패킷 구조 없음

의견에서 "5초마다 ping으로 시간 동기화 + heartbeat"라고 했지만:

- ping/pong 패킷 필드 정의 없음
- heartbeat와 시간 동기화를 같은 패킷으로 묶으면 실패 감지가 결합됨
- §2.2 유휴 타임아웃과의 관계 미정의 (ping 미수신 기준 vs 모든 패킷 미수신 기준)

> **보강 요청**: Ping/Pong 필드 정의, heartbeat 판정 기준, 시간 동기화 결과의 사용처 명시

- 지금은 그냥 핑 쏘면 서버에서 서버시간 날리는걸로. 10초마다 하트비트 판정. 시간 동기화 해야 이동 동기화든 어디든 쓰니까 일단 추가. 경험상임

---

### R-11. 수신 버퍼 16KB = 최대 패킷 16KB

수신 버퍼와 최대 패킷 크기가 동일하면 TCP 스트림 파싱에서 이전 패킷 잔여 + 새 패킷 시작 혼재 처리 불가.

> **보강 요청**: 수신 버퍼 전략(링 버퍼 vs 파싱 상태 머신) 명시. 버퍼 크기를 최대 패킷의 최소 2배로 설정하거나 파싱 전략 확정

- 링 버퍼로 처리. 이러면 크기가 같아도 문제 없음 꽉 차면 파싱 되어야 하니까.

### R-12. §2.1 "asio 사용"만으로는 결정이 아님

completion handler vs coroutine, strand 사용 여부, 세션당 write serialization, lifetime ownership(shared_ptr/intrusive)이 빠져 있어 §2.3 스레딩 논의와 결합되지 못함.

> **보강 요청**: standalone vs Boost 확정, 코루틴 사용 여부, strand 정책까지 함께 결정

- asio standalone이고, 코루틴으로. 코루틴 / completion handler 두개 성능 비교 가능했으면 함 추상화해서.
- 자세한 건 따로 논의

- 위에서 설명함

### R-18. 패킷 직렬화 포맷 관계 불명확

§1.1은 수동 바이트 레이아웃, §1.2는 protobuf 생성기 언급. 헤더는 수동이고 payload만 protobuf인지, 전체가 protobuf인지 관계가 정의되지 않음.

- 프로토콜 버퍼를 패킷 정의에 쓴다고 했지 rpc 쓴다고 안 함. 헤더는 수동
- 아니면 스키마 작성한 거 generator로 패킷에 쓰는 코드 작성

---

### R-19. 클라이언트 수신 스레드 분리 여부

§4.1은 Tick 기반 Pop만 논의. 별도 수신 스레드에서 버퍼링하는지, 언리얼 메인 스레드에서 직접 recv하는지 불명확.

- 송수신 스레드 있고, 언리얼 메인 스레드는 소비만 함
