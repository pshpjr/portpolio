## §1. 프로토콜 계층

### §1.1 패킷 프레이밍 포맷 — ✅ 확정

**무엇을 결정해야 하나**

패킷 하나의 바이트 레이아웃을 확정해야 합니다.
서버와 클라이언트 양측이 이 포맷을 공유하므로, 변경 비용이 큽니다.

**선택지**


|   | 헤더 구조                                    | 길이 필드       | 비고                            |
| - | -------------------------------------------- | --------------- | ------------------------------- |
| A | `[length: 2B][type: 2B][payload…]`          | 페이로드만 포함 | 총 헤더 4B, 최대 페이로드 64 KB |
| B | `[length: 4B][type: 2B][payload…]`          | 페이로드만 포함 | 총 헤더 6B, 최대 4 GB           |
| C | `[length: 4B][type: 2B][seq: 4B][payload…]` | 페이로드만 포함 | 시퀀스 번호 포함, 헤더 10B      |

- length 필드: 페이로드 길이만 포함하는지, 헤더 포함 전체 길이인지
- 바이트 순서: little-endian vs. big-endian (네트워크 표준은 big-endian)
- 최대 패킷 크기 제한 필요 여부 (초과 시 Disconnect vs. Skip)

**권장안**: A 또는 B, little-endian, length = 페이로드만, 최대 크기 64 KB 초과 시 Disconnect.시퀀스 번호(옵션 C)는 TCP가 순서 보장하므로 v1에서 불필요.

> **의견**

- bigendian 쓸 필요 없을 것 같다.
- 길이는 2B면 충분. 패킷 타입도 2B로 될 것 같다. 시퀸스 번호 필요 x
- length는 헤더 포함 전체 길이
- 최대 패킷 크기는 16KB로 처리. 넘어가면 disconnect.
  - disconnect 사유 Enum 생성 필요
    - LargePacket
    - ContentRequest
      - 여기 reason 필요할지?
    - DDoS
    - NetworkError
      - errorcode 필요?

> **결정**: `[length: 2B][type: 2B][payload…]`, little-endian, length = 헤더 포함 전체 길이, 최대 64KB, 초과 시 disconnect. 시퀀스 번호 없음.
>
> - 수신 버퍼: 링 버퍼 방식 (§R-11)
> - disconnect 사유 Enum: `LargePacket`, `ContentRequest`, `DDoS`, `NetworkError` (서버 내부 로그용, 클라이언트에 전송하지 않음 §R-7)
> - 헤더는 수동 바이트 레이아웃, payload는 protobuf 스키마 기반 (§R-18)

### §1.2 패킷 타입 ID 할당 규칙 — 🔴 부분 확정

**무엇을 결정해야 하나**

패킷 타입 ID(`type` 필드)의 숫자 범위를 방향별·도메인별로 나누어 충돌을 방지합니다.

**선택지**


| 범위           | 방안 A (방향 우선) | 방안 B (도메인 우선) |
| -------------- | ------------------ | -------------------- |
| 0x0000–0x0FFF | 서버 → 클라이언트 | 연결/세션 관련       |
| 0x1000–0x1FFF | 클라이언트 → 서버 | 캐릭터/이동          |
| 0x2000–0x2FFF | —                 | 전투                 |
| …             | —                 | …                   |

- 방안 A: 방향이 명확해서 핸들러 등록 실수 감지가 쉽다.
- 방안 B: 관련 패킷을 숫자적으로 묶어두어 protocol.md 읽기가 편하다.

**권장안**: 방안 A (방향 우선). 핸들러 미스매치를 즉시 잡을 수 있음.

> **의견**

- protobuf 생성 시 SC, CS를 따로 생성 가능한지? 혹은 protobuf 바탕으로 핸들러 등록까지 자동으로 된 h 만들고, 이 때 SC는 안 등록하는 방안도 있음.
  - generator는 빠르게 생성해야 할 듯.
  - 로그인 / 로그인 이후... 이런식으로 명시적 핸들러 관리하기로 했는데, 이건 어떻게 할 건지? generate 규칙 추가?
  - 클라이언트용 패킷 핸들러도 있어야 함
- 안 쓰는 프로토콜 경고 주는 스크립트도 있으면 좋다. -> 낮은 우선순위로 큐잉
- ~~보안을 위한 프로토콜 번호 셔플~~ → **v1에서 폐기** (§R-15)
- 패킷 네이밍 규칙은
  - 접두사 SC, CS : 패킷 방향
  - 접미사 REQ, RES : REQ : 요청 RES : 요청 응답, 그 외 (서버에서 먼저 패킷 쏘는것들 ex. 이동동기화)는 안 붙음

> **결정**: 방안 A (방향 우선). 네이밍: SC/CS 접두사 + REQ/RES 접미사. 프로토콜 셔플 폐기. protobuf 생성 규칙·핸들러 등록 구조는 별도 논의 필요.

### §1.3 서버 공통 에러 응답 포맷 — 🔴 부분 확정

> `pending-decisions.md` #18과 동일 항목

**무엇을 결정해야 하나**

서버가 요청을 거부하거나 에러가 발생했을 때 클라이언트에 보내는 공통 패킷 포맷을 확정합니다.
이 포맷이 없으면 각 핸들러가 독자 에러 응답을 만들어 클라이언트 처리가 복잡해집니다.

**제안 구조**

```
ErrorResponse {
  request_type: uint16   // 어떤 요청에 대한 응답인지
  error_code:   uint16   // 에러 종류
  message:      string   // 로그/디버그용 (빈 문자열 허용)
}
```

**에러 코드 후보**


| 코드 | 이름              | 상황                             |
| ---- | ----------------- | -------------------------------- |
| 1001 | INVALID_SESSION   | 세션 미인증 상태에서 요청        |
| 1002 | ENTITY_NOT_FOUND  | 존재하지 않는 EntityId 대상 요청 |
| 1003 | ZONE_MISMATCH     | 현재 구역과 다른 구역 대상 요청  |
| 1004 | ALREADY_CONNECTED | 중복 연결 시도                   |
| 1005 | INVALID_PACKET    | 프레임 불량·디코딩 실패         |

> **의견**

- 에러 코드는 모든 프로토콜에서 다 정의하는 하나의 enum으로 protobuf에 정의하고, RES마다 두는 걸로. ErrorResponse는 패킷이 이상하다던가 하는 특이 상황에만 사용
- disconnect 사유는 클라이언트에 전송하지 않음 (서버 내부 로그 전용)
- "특이 상황" = 컨텐츠와 관련되지 않은 에러 (디코딩 실패, 세션 상태 불일치 등). 구현하며 추가 예정

> **결정**: per-RES 에러 코드(전역 enum) + ErrorResponse(비정상 패킷 전용). disconnect 사유는 클라이언트 미전송. 에러 코드 전체 목록은 구현 시 확정.

## §2. 서버 네트워크 구현

### §2.1 서버 I/O 모델 — ✅ 확정

**무엇을 결정해야 하나**

TCP 서버의 I/O 처리 방식을 결정합니다. 이 결정이 전체 서버 코드 구조를 결정합니다.

**선택지**


|   | 방식                                | 장점                               | 단점                               |
| - | ----------------------------------- | ---------------------------------- | ---------------------------------- |
| A | **Asio (Boost.Asio / standalone)**  | 이식성, 코루틴 지원, 성숙한 생태계 | 외부 라이브러리 의존               |
| B | **플랫폼 직접 (epoll/IOCP)**        | 의존성 없음, 구현 제어권 최대      | 코드량 많음, 이식성 없음           |
| C | **libuv**                           | 크로스플랫폼, Node.js 검증         | C API, C++ 래핑 필요               |
| D | **단순 blocking + 스레드-per-세션** | 가장 단순                          | 세션 수 제한, 포트폴리오 가치 낮음 |

**포트폴리오 관점**: 비동기 I/O(A 또는 B)를 선택하는 것이 게임 서버 역량을 더 잘 보여줍니다.**권장안**: Asio standalone (헤더 전용, CMake 의존성 추가 minimal). vcpkg에 이미 있음.

> **결정**: Asio standalone, C++20 코루틴 기반. completion handler와 코루틴 성능 비교 가능하도록 추상화. 상세 설계(strand 정책, lifetime ownership 등)는 별도 논의.

### §2.2 서버 세션 관리 — ✅ 확정

**무엇을 결정해야 하나**


| 항목                | 결정 필요 내용                                    |
| ------------------- | ------------------------------------------------- |
| 최대 동시 연결 수   | v1 목표 (예: 5~10명, 포트폴리오 규모)             |
| 세션 풀             | 미리 할당 vs. 동적 생성                           |
| 수신 버퍼 크기      | 고정 vs. 동적 (예: 64 KB 고정)                    |
| 핸드셰이크 타임아웃 | 연결 후 로그인 패킷이 N초 내 미도착 시 Disconnect |
| 유휴 타임아웃       | 패킷 미수신 상태 지속 시 Disconnect               |

**권장안 초안**

- 최대 연결 수: 10 (포트폴리오 규모)
- 수신 버퍼: 64 KB 고정
- 핸드셰이크 타임아웃: 5초
- 유휴 타임아웃: 30초 (또는 없음, 클라이언트가 keepalive 담당)

> **의견**

- 최대 연결 2000
- 수신 버퍼 16KB
- 로그인 타임아웃 3초
- 클라는 5초마다 ping 쏴서 서버 클라 시간 동기화
- 세션 풀 사용

> **결정**:
>
> - 최대 연결: 2000
> - 세션 풀: 미리 할당
> - 수신 버퍼: 16KB 링 버퍼
> - 핸드셰이크 타임아웃: 3초
> - heartbeat: 클라이언트 5초마다 ping → 서버가 서버 시간 응답. 10초마다 heartbeat 판정
> - 송신 큐: backpressure 시 session disconnect + 패킷 drop

### §2.3 서버 스레딩 모델 — 🔴 부분 확정

**무엇을 결정해야 하나**

I/O 처리와 게임 로직이 같은 스레드에서 돌아가는지, 분리되는지를 결정합니다.

**선택지**


|   | 구조                                         | 비고                        |
| - | -------------------------------------------- | --------------------------- |
| A | I/O 스레드 (1개) + 게임 로직 스레드 (1개)    | 잠금 범위 명확, 단순        |
| B | I/O 스레드 풀 (N개) + 게임 로직 스레드 (1개) | 높은 처리량, 동기화 복잡    |
| C | 단일 스레드 (I/O + 로직 모두)                | 동기화 없음, 단일 코어 병목 |

**권장안**: A. 포트폴리오 규모에서 I/O 병목이 없고, 스레드 경계가 명확해 코드 리뷰가 쉽다.

> **의견**

- IO 스레드 풀 + 컨텐츠 스레드 풀
- io 스레드 풀은 io bound 높으니 concurrency 높게. 컨텐츠 스레드는 cpu bound 높으니 낮게
- connection ID 기반 해서 modular 하는 걸로 처리 해당 스레드 큐에 push / 만약 MO 던전 들어갔으면 그 던전에 push
- 던전은 던전 스레드 풀에서 비동기 처리
- io풀 / 던전 풀 / 던전 밖 세션 풀 로 관리
- io 풀에서  DB도 처리할지는 미정.

> **결정**:
>
> - IO 스레드 풀(`hardware_concurrency`) + 로직 스레드 풀(`0.5 * hardware_concurrency`) + DB 스레드 풀(`0.5 * hardware_concurrency`) + 던전 스레드 풀
> - IO→로직: connectionID modular → 담당 스레드의 MPSC 큐에 push. 던전 입장 시 던전 큐로 전환
> - 단일 TCP 소켓에 멀티스레드 send 허용
> - sendQ backpressure → session disconnect + 패킷 drop
> - **미결**: 던전 입장/퇴장 시 큐 전환 원자성 보장 방식, DB callback 처리 방식, 던전 풀 스레드당 인스턴스 상한

## §3. 프로토콜 / 엔티티 설계

### §3.1 EntityId 고유성 범위와 시간 동기화 — ✅ 확정

> `pending-decisions.md` #17과 동일 항목

**무엇을 결정해야 하나**


| 항목                 | 선택지                                  |
| -------------------- | --------------------------------------- |
| EntityId 고유성 범위 | 서버 프로세스 전역 vs. 던전 인스턴스 내 |
| EntityId 재사용      | 세션 종료 후 재사용 허용 여부           |
| MoveTimestamp 기준   | 서버 클록 vs. 클라이언트 로컬           |
| 시간 오프셋 동기화   | v1 포함 여부                            |

**권장안**

- EntityId: 서버 전역 단조 증가 uint64 (재사용 없음, 오버플로 사실상 불가)
- MoveTimestamp: 서버 클록 기준, 세션 시작 후 경과 ms (int64). 클라이언트가 `ConnectedAt`을 기록해 오프셋을 계산.
- 시간 동기화: v1은 서버 타임스탬프를 그대로 사용, 오프셋 보정은 v2.

> **의견**

- EntityID는 던전 인스턴스 내 고유. 필드에 있는 actor의 id는 인스턴스마다 중복될 수 있음.
- 클라는 로그인 시 서버 시간을 받아오며, 5초마다 ping 패킷으로 시간 동기화 + heartbeat 체크한다.

> **결정**:
>
> - EntityId: uint64, 상위 32비트에 객체 타입 등 메타 정보. 인스턴스 내 고유 (인스턴스 간 중복 가능). 재사용 없음.
> - 이전 구역은 추후 별도 프로세스(scale-out). 로그에는 EntityID + 타입/닉네임 함께 기록.
> - 시간 동기화: 로그인 시 서버 시간 수신, 5초마다 ping → 서버 시간 응답.

### §3.2 엔티티 스폰 패킷 유형 구분 — ✅ 확정

> `pending-decisions.md` #14와 동일 항목

**무엇을 결정해야 하나**


| 항목                    | 선택지                                     |
| ----------------------- | ------------------------------------------ |
| EntityType 구분 방법    | enum 필드 vs. TemplateId 범위 규칙         |
| 몬스터 전용 필드        | 공통 스폰 패킷 optional vs. 별도 패킷 타입 |
| 원격 플레이어 장비 외형 | 스폰 패킷 포함 vs. 별도 패킷               |

**권장안**

- EntityType enum 도입: `PLAYER=0, MONSTER=1, NPC=2`. TemplateId 범위 규칙은 암묵적이고 검증이 어렵다.
- 몬스터 전용 필드(무력화 게이지 등)는 별도 `MonsterSpawnPacket`으로 분리. 공통 패킷이 커지는 것을 방지.
- 원격 플레이어 장비 외형: v1은 스폰 패킷에 TemplateId만. 외형 정보는 v2.

> **의견**

- EntityType Enum 전달. remote player의 외형 정보도 같이 전달한다. 외형 정보를 원하는 remote player id 배열을 서버에 전달하면, 각각 정보를 주는 방식

> **결정**:
>
> - EntityType enum 사용.
> - 스폰 패킷: 기본 외형(종족, 이름) 포함. 디테일 외형은 클라이언트가 LOD 방식으로 단계별 요청.
> - 외형 변경(장비 교체 등): 클라이언트가 로컬 처리 (서버에 재요청하지 않음).
> - 보스 전용 필드: 별도 구조체/패킷으로 분리.

### §3.3 구역 전환 프로토콜 — 🔴 부분 확정

> `pending-decisions.md` #12, #15와 동일 항목

**무엇을 결정해야 하나**


| 항목                      | 선택지                                               |
| ------------------------- | ---------------------------------------------------- |
| 전환 시작 패킷 이름       | `ZoneTransitionBegin`, `DungeonZoneChange`, …       |
| 클라이언트 응답 필요 여부 | 요청-응답 왕복 vs. 서버 단방향 Push                  |

**권장안**

- `ZoneTransitionBegin { zone_id }` → 클라이언트 페이드 아웃 + 일괄 디스폰 + 플레이어 입력 차단

- `ZoneTransitionEnd { zone_id }` → 클라이언트 페이드 인 + 입력 해제
- 전환 중 수신된 이전 구역 패킷: EntityId Map에 없으므로 무시 (기존 "미등록 EntityId 무시" 규칙으로 커버)

> **의견**

- 지금 패킷 이름 정하는 시기 아님

> **결정**:
>
> - 서버 단방향 Push. 클라이언트 ACK 불필요.
> - 전환 중 클라이언트에 "전환중" 플래그 필요. 전환 타임아웃 30초.
> - 패킷 이름은 구현 단계에서 확정.
> - **미결**: 전환 중 클라이언트 상태 전이 상세, 보존/초기화 상태 경계

> **결정**:

> - WorldEntitySubsystem이 자체 Tick을 가질 수 있음 (같은 게임 스레드).
> - **미결**: 구체적인 TickGroup, Tick 순서 보장 방식은 구현 단계에서 확정
