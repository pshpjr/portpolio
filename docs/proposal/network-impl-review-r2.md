### A-1. 스레딩 모델

**현재**: IO풀(`HC`) + 로직풀(`0.5×HC`) + DB풀(`0.5×HC`) + 던전풀


|                  | 현재(4풀) | Strand-per-Session | IO+로직 2스레드 |
| ---------------- | --------- | ------------------ | --------------- |
| 구현 복잡도      | 매우 높음 | 중간               | 낮음            |
| 동시성 버그 위험 | 높음      | 낮음               | 거의 없음       |
| 2000 동접 처리   | 가능      | 가능               | 가능            |
| 던전 격리        | 물리 분리 | strand 분리        | 시분할          |
| scale-out 준비   | 큐만 교체 | 큐만 교체          | 아키텍처 변경   |
| 포트폴리오 어필  | "풀 설계" | "strand 활용"      | "단순함"        |

**3자 권장**: v1은 IO풀 + 로직 단일(또는 strand) + DB풀로 시작.

> **답변**: 멀티스레드 환경 동시성 관리는 포트폴리오의 핵심 중 하나입니다. 또한, strand 안 쓸겁니다. 그건 포트폴리오에 의미가 없어요. 단순하니까 이걸 어필하라고요?

**후속 논의**: 4풀 유지가 확정되었으므로 질문을 전환한다. "4풀을 쓸지 말지"가 아니라 **"4풀에서 R-1의 미해소 계약을 어떻게 채울지"**가 남은 과제다.

R-1에서 빠진 계약 중 현재 답변된 것과 미답변:


| 계약                 | 상태      | 내용                                                      |

> **답변**: 단조 증가하면 전역 유일하지 않죠. 아니 애초에 인스턴스 간 고유하게 ID를 어떻게 부여하려고요. 다른 프로세스로 쪼갤거라니까요?

### A-6. 문서에서 누락된 패턴

> **의견**: 문서가 너무 짧아요. 뭘 얘기하는건진 알려줘야죠. AOI는 필수. single-writer는 atomic flag. 패킷 배치 전송은 성능 개선 요소. 메트릭은 기본.

#### 1. AOI (Area of Interest) / 브로드캐스트 구조 — **v1 필수**

서버가 "이 패킷을 누구에게 보낼 것인가"를 결정하는 공간 필터링 시스템. 200명이 한 구역에 있을 때 모든 이동 패킷을 200명 전원에게 보내면 초당 `200 × 200 × 20Hz = 800K` 패킷. AOI가 없으면 동접 50명만 넘어도 대역폭이 폭발한다.

- **격자 기반 AOI**: 월드를 고정 크기 셀로 나누고, 인접 셀의 엔티티에게만 브로드캐스트. 구현 단순, O(1) 셀 조회
  - 좌표 -> 셀 변환. 셀을 hashing 해서 Hashmap에 각 객체 리스트 저장. 배열보다 메모리 최적화
- **반경 기반 AOI**: 엔티티 중심 반경 내 대상만. 정밀하지만 매 Tick마다 거리 계산 필요
  - 매틱이 아니라 좀 더 드물게 갱신해도 되는지?
- **하이브리드**: 격자로 1차 필터 → 반경으로 2차 필터
- **결정 필요**: 셀 크기, 갱신 주기, AOI 진입/이탈 이벤트 패킷 정의, 구역 경계 처리

#### 2. Send 직렬화 (single-writer per socket) — **v1 필수**

멀티스레드에서 하나의 TCP 소켓에 동시에 `send()`를 호출하면 패킷 바이트가 섞이는 corruption 발생. 직렬화 방법:

- **atomic flag + spin**: send 전에 flag를 acquire, 완료 후 release. 경합 시 spin-wait. 구현 단순, 짧은 send에 적합
  - spin이라고 말한 적 없음. mpsc 큐에 패킷 push 후 flag acqure. 실패 시 그냥 리턴, 획득 시 io_context에 send post 후 리턴
- **MPSC 송신 큐 + 전담 flush 스레드**: 보낼 패킷을 큐에 넣고, 전담 스레드가 순서대로 send. 경합 없음, 배치 전송과 자연스럽게 결합
- **strand**: Asio의 strand로 직렬화. 추가 코드 없음, 단 Asio에 의존

> **결정 방향**: atomic flag 사용 예정

#### 3. 패킷 배치 전송 (Packet Coalescing) — **v1 성능 개선으로 포함 예정**

같은 Tick에 한 세션에게 보낼 패킷 여러 개를 모아서 한 번의 시스템콜(`writev` / scatter-gather)로 전송. 개별 `send()`마다 커널 전환이 발생하므로, 배치로 묶으면 시스템콜 횟수가 `N → 1`로 줄어든다.

- 200명에게 이동 패킷 브로드캐스트: 개별 send = 200 syscall, 배치 = 200 세션 × 1 syscall
- 구현: 송신 큐에 Tick 단위로 적재 → Tick 끝에 flush
- send 직렬화(#2)와 결합 시 MPSC 큐 + flush 모델이 자연스러움
- scatter-gather 자체도 비용이 있으므로, 저장된 패킷을 합칠 수 있으면 memcpy 처리

#### 4. Rate Limiting — **v1 필수**

악의적 또는 버그 클라이언트가 초당 수천 개의 패킷을 보내면 서버 로직 스레드가 마비된다. 패킷 타입별로 허용 빈도를 제한해야 한다.

- **예시**: 이동 입력 20/s, 스킬 사용 5/s, 채팅 3/s, 외형 요청 1/s
- 초과 시: silent drop 또는 경고 후 disconnect
- 구현: 세션별 토큰 버킷 또는 슬라이딩 윈도우 카운터
- 그냥 전체 수신 패킷 개수로 처리. 초당 몇 개 인지

#### 5. 프로토콜 버전 교환 — **v1 권장**

클라이언트와 서버의 빌드 버전이 다르면 패킷 구조가 달라 조용한 디코딩 실패가 발생한다. 핸드셰이크 시 프로토콜 버전을 교환하고, 불일치 시 명시적으로 연결을 거부해야 한다.

- 로그인 패킷에 `protocol_version: uint16` 필드 추가
- 서버가 지원하지 않는 버전이면 에러 응답 후 disconnect

#### 6. 메트릭 / 텔레메트리 — **v1 필수**

런타임에 서버 상태를 수치로 관찰하는 계측 시스템. 성능 병목을 코드 수정 없이 발견하려면 처음부터 있어야 한다.

- **수집 대상 예시**: 패킷 처리 시간(평균/p99), 큐 깊이(IO→로직, 송신), RTT 분포, 세션 수, 초당 패킷 수, 메모리 사용량
- **구현**: 카운터/히스토그램을 코드에 삽입 → 주기적으로 로그 또는 외부 시스템에 전송
- 포트폴리오에서는 Prometheus/Grafana까지 안 가도, 서버 콘솔에 주기적으로 출력하는 것만으로도 가치 있음
- 콘솔 로깅. 이를 위한 시스템 필요

#### 7. RTT 측정 — **v1 권장**

Ping 패킷에 클라이언트 전송 시각을 실어 보내고, 서버가 서버 시간 + 수신 시각을 응답하면 클라이언트가 왕복 시간(RTT)을 계산할 수 있다.

- 이동 보간, 스킬 판정 보정, 네트워크 품질 표시에 활용
- §2.2의 5초 Ping에 이미 시간 동기화를 태우고 있으므로, RTT 계산 필드만 추가하면 됨

#### 8. Graceful Shutdown — **v1 권장**

서버 종료 시 접속 중인 세션에 사전 알림 없이 TCP를 끊으면 클라이언트가 "네트워크 에러"를 표시한다. 종료 전에 공지 패킷을 보내고, 일정 시간 후 세션을 순차 정리해야 한다.

- 순서: 신규 접속 거부 → 전체 세션에 종료 예고 패킷 → 대기(5초) → 세션 정리 → 풀 drain → 프로세스 종료
- 필요한 것이 맞음

#### 9. request_id / correlation_id — **v1 권장**

REQ 패킷에 클라이언트가 부여한 고유 ID를 실어 보내고, 서버가 RES에 그대로 돌려준다. 비동기 응답이 어떤 요청에 대한 것인지 매칭할 수 있고, DB callback 추적, 타임아웃 로그에도 유용하다.

- 클라 입장에서 꼭 필요할지 확인 필요.

#### 10. heartbeat / time sync 분리 — **v1 권장**

현재 5초 Ping이 시간 동기화 + heartbeat를 동시에 담당한다. 기능적으로는 문제없지만, heartbeat 실패 판정 기준(10초)과 시간 동기화 정밀도가 결합되어 있어 나중에 어느 한쪽만 조정하기 어렵다.

- 분리하면: heartbeat는 빈 패킷(1~2B), time sync는 별도 주기(30초)로 분리 가능
- 현재 구조 유지도 가능 — 결합의 단점이 실제로 문제될지는 구현 시 판단
- 문제 안 될거라 판단

#### 11. 서버 간 메시지 인터페이스 — **v1 인터페이스만**

v1은 단일 프로세스지만, scale-out 시 마을 서버↔던전 서버 간 통신이 필요하다. v1에서 모듈 간 호출을 함수 호출이 아닌 메시지 패싱 인터페이스로 설계해두면, v2에서 네트워크 전송으로 교체할 때 호출부를 바꿀 필요가 없다.

#### 12. snapshot chunking — **v2**

구역 전환 시 대량 엔티티 스폰 데이터를 한 패킷에 담을 수 없을 때(64KB 초과), 여러 청크로 나눠 전송하는 프로토콜. v1은 LOD 방식으로 스폰 데이터를 줄여 64KB 내에 들어올 가능성이 높으므로 v2로 미룸.

#### 13. 패킷 우선순위 큐 — **v2**

이동 동기화(20Hz, 지연 민감)와 채팅(1회성, 지연 둔감)이 같은 송신 큐에 있으면, 큐가 밀릴 때 이동 패킷이 채팅 뒤에서 대기. 우선순위 큐 또는 채널 분리로 해결. v1은 단일 큐로 시작.

## 열린 쟁점 해법 — Codex + OpenCode 교차 비교 (2026-04-08)

> 6개 열린 쟁점에 대해 Codex(GPT-5.4)와 OpenCode(GLM-5.1)가 독립적으로 해법을 제안한 결과입니다.

### 합의 (1건)

#### 쟁점 1. 세션 FSM — 2계층 분리 확정

**세션 설계안 반영 (2026-04-08)**: FSM을 네트워크 레이어와 게임 레이어로 분리한다.

**네트워크 레이어 FSM** (Session 클래스 소유, 4상태):

```
Negotiating → Active → Closing → Closed
                 └──→ Closed (비정상 종료)
Negotiating → Closed (타임아웃/에러)
```

- `Negotiating`: Accept 완료, 프로토콜 판별 진행 중. OnConnect 호출 전이므로 컨텐츠가 존재를 모름
- `Active`: 프로토콜 확정, 정상 통신 가능
- `Closing`: sendClose() 호출됨, send queue drain 후 종료
- `Closed`: 종료 완료
- 해당 사항 tcp, http 다른 포트 쓰기로 하면서 negotiating 필요 없음. 그냥 add acceptor 형태의 인터페이스로 처리.

**게임 레이어 FSM** (컨텐츠 소유, 네트워크 Active 상태 내에서 동작):

```
Authenticated → InLobby → InDungeon → Transitioning
```

- `InLobby/InDungeon` 분리: 허용 패킷이 완전히 다르고, 큐 바인딩(로직 스레드 ↔ 던전 스레드)과 1:1 매핑
- `Transitioning`: 구역 전환 중. 월드 조작 패킷 거부
- **불허 패킷**: ErrorResponse 후 유지 → 임계치(3~5회) 초과 시 disconnect. rate limiter와 카운터 공유
- connected 추가 필요. 또한, 지금처럼 순차 진행이 아님.

이 분리로 네트워크 세션은 게임 규칙을 모르고, 게임 레이어는 TCP 상태를 모른다.

### 세션 설계안 반영 (3건 확정, 1건 등록)

#### 쟁점 7. send 직렬화 — 확정

**세션 설계안 반영 (2026-04-08)**: atomic_flag + moodycamel ConcurrentQueue + flush 패턴으로 확정.

- `send(buffer)`: Active 상태에서만 큐에 enqueue. 그 외 상태에서 호출 시 drop
- enqueue 후 `isSending` atomic_flag가 false이면 set하고 flush 시작
- flush가 큐를 drain하다 비면 flag clear
- Closing 상태에서는 큐에 남은 데이터를 계속 flush, 비거나 I/O 에러 시 Closed 전이
- 배치 전송(writev/scatter-gather)은 flush 루프에서 자연스럽게 결합 가능

#### 쟁점 8. 종료 처리 — close/sendClose strand 직렬화로 변경

**변경 (2026-04-08)**: CAS 기반 → strand 직렬화로 전환.

- close(), sendClose(), recv completion, 상태 전이 → 모두 strand에서 실행
- strand가 직렬화를 보장하므로 CAS 불필요, 상태 전이 로직이 단순해짐
- **미정**: 서버 전체 종료 시 4풀(IO/로직/DB/던전)의 drain 순서

#### 쟁점 9. 세션 종료 라이프사이클 — 논의 대기

close/sendClose/Negotiating 종료의 상세 시퀀스, 콜백 호출 순서, I/O 에러 조합 시나리오를 정리해야 한다.
세션 설계안([session-class-architecture.md](session-class-architecture.md))에 초안이 있으나, strand 모델 변경에 맞춰 재정리 필요.

#### 쟁점 10. 세션 동시성 모델 — strand 허용으로 변경

**변경 (2026-04-08)**: 포트폴리오 가독성을 위해 strand 사용을 허용한다. send만 atomic flag로 최적화.


| 관심사                   | 이전 (CAS only)                 | 변경 후 (strand + atomic flag)    |
| ------------------------ | ------------------------------- | --------------------------------- |
| 상태 전이                | atomic CAS                      | strand 직렬화                     |
| OnDisconnect 단일 호출   | CAS 성공 스레드만 호출          | strand 내에서 1회 호출 보장       |
| recv completion          | 별도 스레드에서 실행, 상태 확인 | strand에서 실행                   |
| send 큐 enqueue          | ConcurrentQueue (변경 없음)     | ConcurrentQueue (변경 없음)       |
| send flush (async_write) | atomic_flag isSending           | atomic_flag isSending (변경 없음) |
| close/sendClose          | CAS + socket::close()           | strand::post + socket::close()    |

**변경 이유**: CAS 패턴은 정확하지만 읽는 사람이 모든 경합 시나리오를 머릿속에서 재구성해야 한다. strand를 쓰면 "이 핸들러들은 동시에 실행되지 않는다"는 보장이 코드 구조에서 바로 보인다. send만 atomic flag를 유지하는 이유는 enqueue가 strand 밖(임의 스레드)에서 호출되기 때문이다.
