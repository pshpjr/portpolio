# Server-Proposal Review: client/network_subsystem.md

> 검토 대상: `docs/proposal/client/network_subsystem.md`
> 검토 날짜: 2026-04-06
> 검토 관점: 서버 구현자가 기획서에서 찾아야 할 계약 누락

---

## 1. 스폰 패킷 필드 누락

### 1-1. 엔티티 유형 구분자 부재

`FClientRemoteCharacterState`에 `EntityId`와 `TemplateId`가 있으나, 이 엔티티가 **플레이어인지 몬스터인지 NPC인지**를 구분하는 필드가 없다.

**기획서가 추가해야 할 것:**
- `EntityType` 열거형 (예: `PLAYER`, `MONSTER`, `NPC`, `GIMMICK_OBJECT`) 을 `FClientRemoteCharacterState` 테이블에 명시하고, 서버 스폰 패킷의 필수 필드로 정의할 것.
- `TemplateId`만으로 유형을 역추론하는 구조라면, TemplateId 대역 규칙(예: 1xxxx = 플레이어, 2xxxx = 몬스터)을 기획서에서 확정할 것.

### 1-2. 몬스터/NPC 스폰이 원격 캐릭터와 동일 흐름을 공유하는지 불명확

기획서는 "나 외의 엔티티(다른 플레이어, 몬스터, NPC)"를 하나의 `RemoteCharacterSpawn` 패킷 흐름으로 묶고 있지만, 미결 항목에서 "몬스터/NPC 전용 타입이 필요하면 구현 단계 결정"이라 남겨뒀다.

**기획서가 추가해야 할 것:**
- 서버가 몬스터를 스폰할 때 보내야 하는 추가 정보 목록을 확정할 것. 예를 들어:
  - `StaggerGauge` / `MaxStaggerGauge` (보스/정예): 무력화 게이지 UI가 필요한 대상인지 여부
  - `BossPhase`: 보스인 경우 현재 페이즈
  - `AIState` 또는 `PatternId`: 접속 시점에 이미 진행 중인 보스 패턴이 있다면 초기 상태
  - `ElementType`: 속성 상호작용을 위한 속성 종류
- 이 필드들이 원격 플레이어 스폰과 동일 패킷에 optional로 들어가는지, 별도 패킷 타입(`MonsterSpawn`, `NpcSpawn`)으로 분리되는지 기획 수준에서 결정할 것.

### 1-3. 플레이어 스폰 시 장비/외형 정보 누락

원격 플레이어의 스폰 패킷에 무기 종류, 장비 외형, 무기 프리셋 정보가 없다. 다른 플레이어의 외형을 렌더링하려면 서버가 어떤 외형 데이터를 스폰 시점에 보내야 하는지 정의되어야 한다.

**기획서가 추가해야 할 것:**
- 원격 플레이어 스폰 패킷에 포함해야 할 외형 관련 필드 목록 (예: `WeaponType`, `EquippedVisuals[]` 등)
- 장비 변경 시 외형 갱신 패킷이 별도로 필요한지 여부

## 2. 서버 권위 상태 전이 중 클라이언트 처리 누락 케이스

### 2-1. HP 변동/사망/부활의 원격 엔티티 반영 경로 미정의

`death_revival.md`는 사망, 부활, 내구도 감소를 상세히 정의하지만, `network_subsystem.md`의 `FClientRemoteCharacterState`는 `bIsDead` 플래그와 `CurrentHP`만 있다.

**기획서가 추가해야 할 것:**
- 서버가 원격 엔티티의 사망을 알릴 때의 패킷 구조 (사망 이유 포함 여부, 부활 대기 상태 표시)
- 서버가 원격 엔티티의 부활을 알릴 때 전달해야 할 필드 (부활 위치, 부활 후 HP, 보호 시간 잔여량)
- `ReviveStateEvent`(protocol.md에 정의됨)와 `FClientRemoteCharacterState` 갱신의 매핑 관계

### 2-2. 상태 이상(Status Effect) 동기화 경로 부재

`combat.md`에서 감전/중독/화상/빙결 4종 상태 이상을 정의하지만, 원격 엔티티에 현재 적용 중인 상태 이상을 전달하는 필드가 `FClientRemoteCharacterState`에 없다.

**기획서가 추가해야 할 것:**
- `ActiveStatusEffects[]` 또는 비트마스크 필드를 상태 구조체에 추가하거나, 상태 이상 적용/해제를 별도 이벤트로 정의할 것
- 각 상태 이상의 잔여 지속시간을 클라이언트에 보내는지, 아니면 시작 시점만 보내고 클라이언트가 지속시간을 데이터 테이블에서 조회하는지 결정할 것

### 2-3. 전투 판정 피드백 이벤트의 수신 주체 불명확

`protocol.md`에 `CombatFeedbackEvent`, `InterruptSuccessEvent`, `StaggerStateEvent` 등이 있으나, 이것이 `WorldEntitySubsystem`을 경유하는지, 별도 `CombatSubsystem`이 받는지 기획서에 정의가 없다.

**기획서가 추가해야 할 것:**
- `OnPacketReceived`에서 전투 관련 이벤트를 누가 구독하는지의 라우팅 계약
- 특히 `BossPatternEvent`는 UI(상단 보스 HP/무력화 게이지)와 월드 연출 양쪽에 영향을 주므로, 라우팅 대상을 명시할 것

## 3. 구역 전환 계약 불명확

### 3-1. 구역 전환 프로토콜의 시작 신호 미정의

기획서에는 "새 구역 진입 패킷 수신 시 현재 Map의 모든 엔티티를 일괄 디스폰한다"고만 되어 있다.

**기획서가 추가해야 할 것:**
- 서버가 보내는 구역 전환 시작 패킷의 이름과 필수 필드 (최소: `NewZoneId`, `SpawnPosition`)
- 구역 전환 중 클라이언트가 입력을 차단해야 하는지, 로딩 화면을 보여야 하는지의 상태 정의
- 구역 전환 완료 조건: 클라이언트가 "새 구역 준비 완료"를 서버에 보내야 하는지, 아니면 서버가 일방적으로 스폰 패킷을 보내기 시작하는지
- 구역 전환 중에 수신된 이전 구역의 잔여 패킷을 폐기하는 규칙 (예: ZoneId가 다르면 무시)

### 3-2. 일괄 디스폰 후 스폰 순서 보장

"서버가 새 구역의 스폰 패킷을 순차 발송한다"고 했으나 순서 보장 규칙이 없다.

**기획서가 추가해야 할 것:**
- 새 구역의 스폰 패킷이 모두 도착했음을 클라이언트가 알 수 있는 종료 신호 (예: `ZoneReadyEvent`) 존재 여부
- 스폰 순서에 의존성이 있는지 (예: 보스가 반드시 마지막에 스폰되어야 하는지)
- 스폰 패킷 수신 중 클라이언트가 렌더링을 시작해도 되는 시점

## 4. 재연결 / 세션 재개 계약 부재

### 4-1. 재연결 시 월드 상태 복구 프로토콜 없음

`login.md`에서 "비정상 종료 후 재접속 시 동일 토큰으로 복구를 시도한다"고 했으나, `network_subsystem.md`에는 재연결 관련 계약이 전혀 없다. 연결 상태 전이가 `Disconnected -> Connecting -> Connected`뿐이고, `Reconnecting` 상태가 없다.

**기획서가 추가해야 할 것:**
- 재연결 성공 시 서버가 보내야 하는 월드 스냅샷의 범위 정의 (현재 구역의 모든 엔티티를 다시 스폰하는지, 마지막 확인점부터의 델타만 보내는지)
- 재연결 시 세션 토큰 유효성 검증 후 캐릭터를 어디에 배치하는지 (던전 내 재연결이면 현재 구역? 입구?)
- 재연결 가능 시간 제한 (세션 유효 시간 30분과의 관계)
- 던전 전투 중 재연결 시 파티원에게 어떤 상태가 보이는지 (연결 끊긴 캐릭터의 서버 측 처리: 즉시 제거? 일정 시간 유지?)

### 4-2. 연결 끊김 시 서버 측 캐릭터 처리 계약 없음

`OnDisconnected(reason)`이 발생하면 클라이언트는 UI를 보여주겠지만, 서버 측에서 해당 캐릭터를 어떻게 처리하는지에 대한 계약이 기획 어디에도 없다.

**기획서가 추가해야 할 것:**
- 던전 내 연결 끊김 시 서버가 해당 캐릭터를 유지하는 시간 (grace period)
- grace period 동안 해당 캐릭터의 서버 상 행동 (무적? AI 대리? 즉시 사망?)
- grace period 초과 시 파티원에게 보내는 디스폰 사유 (`DisconnectReason` 필드)
- 파티 전원 연결 끊김 시 던전 인스턴스 유지/파기 규칙

## 5. 이벤트 순서 / 재시도 / 중복 처리

### 5-1. 패킷 순서 역전 대응 규칙 없음

TCP를 사용하므로 전송 순서는 보장되지만, `MoveTimestamp`를 사용하는 이상 기획서는 순서 보장 가정을 명시해야 한다.

**기획서가 추가해야 할 것:**
- TCP 전제 하에서 패킷 순서 보장을 가정한다는 명시적 선언
- 만약 향후 UDP나 멀티 커넥션을 고려한다면, `MoveTimestamp` 기반의 out-of-order 폐기 규칙 정의

### 5-2. 중복 스폰/디스폰 패킷 수신 시 행동 미정의

서버 버그나 구역 전환 경계에서 동일 `EntityId`에 대한 스폰 패킷이 중복 도착할 수 있다.

**기획서가 추가해야 할 것:**
- 이미 존재하는 `EntityId`로 스폰 패킷이 오면: 기존 것을 교체하는지, 무시하는지, 에러 로그를 남기는지
- 존재하지 않는 `EntityId`로 디스폰 패킷이 오면: 무시하는지, 에러 로그를 남기는지
- 존재하지 않는 `EntityId`로 이동 패킷이 오면: 무시하는지, 스폰을 요청하는지

## 6. 저장해야 하는 데이터와 키 구조

### 6-1. EntityId의 생명주기와 재사용 규칙 미정의

`EntityId`는 `uint64`로 "서버 부여 식별자"라고만 되어 있다.

**기획서가 추가해야 할 것:**
- EntityId는 서버 프로세스 생존 기간 동안 고유한지, 던전 인스턴스 내에서만 고유한지
- EntityId는 엔티티 소멸 후 재사용 가능한지 (재사용되면 이전 엔티티에 대한 지연 패킷이 잘못 적용될 수 있음)
- 내 캐릭터의 EntityId도 동일 체계를 사용하는지, 별도인지

### 6-2. 서버 시간 동기화 계약 없음

`MoveTimestamp`가 "이동 패킷 수신 시각 (보간 기준점)"이라 되어 있으나, 이것이 서버 시간인지 클라이언트 시간인지, 서버-클라이언트 시간 오프셋을 어떻게 맞추는지 정의가 없다.

**기획서가 추가해야 할 것:**
- `MoveTimestamp`의 시간 기준 (서버 클록 기준인지 클라이언트 로컬 시각인지)
- 서버-클라이언트 시간 동기화 메커니즘 (ping-pong 기반 오프셋 계산 등)의 존재 여부와 계약
- 보간 알고리즘이 이 타임스탬프에 의존하므로, 정밀도 요구 수준 (ms 단위? 서버 틱 단위?)

## 7. 실패 응답과 복구 조건

### 7-1. 서버 거부 응답의 클라이언트 처리 미정의

클라이언트가 `Send(packet)`으로 요청을 보내지만, 서버가 거부할 때의 응답 형식과 클라이언트 처리가 없다.

**기획서가 추가해야 할 것:**
- 서버 거부/에러 응답의 공통 포맷 (예: `ErrorResponse { requestType, errorCode, message }`)
- 주요 에러 코드 열거 (최소한 `INVALID_SESSION`, `ENTITY_NOT_FOUND`, `ZONE_MISMATCH`, `COOLDOWN_ACTIVE`)
- 에러 응답 수신 시 클라이언트의 기본 행동 (UI 표시? 재시도? 강제 동기화 요청?)

### 7-2. 송신 실패와 연결 끊김 사이의 중간 상태 없음

기획서에 "v1은 에러 로그 + Disconnect"라고 되어 있지만, 단일 패킷 송신 실패로 즉시 연결을 끊는 것은 과도할 수 있다.

**기획서가 추가해야 할 것:**
- 송신 실패의 정의 (TCP write 에러? 타임아웃?)
- 송신 실패 시 즉시 Disconnect가 맞는지, 아니면 N회 재시도 후 Disconnect인지 v1 정책을 확정할 것

---

## 8. protocol.md와의 정합성 갭

### 8-1. 패킷 타입 목록 불일치

`protocol.md`에는 `ActorStateEvent`, `BossPatternEvent`, `CombatFeedbackEvent` 등 다양한 전투 이벤트가 정의되어 있으나, `network_subsystem.md`는 `RemoteCharacterSpawn/Move/Despawn` 3종만 다룬다.

**기획서가 추가해야 할 것:**
- `protocol.md`의 메시지 그룹과 `network_subsystem.md`의 패킷 핸들러 간 매핑 테이블
- `WorldEntitySubsystem`이 처리하지 않는 패킷(전투 피드백, 던전 라이프사이클 등)은 어떤 서브시스템이 처리하는지 라우팅 계약

### 8-2. DungeonLifecycle 이벤트와 구역 전환의 연결 부재

`protocol.md`에 `DungeonEnterEvent`, `CheckpointReachedEvent`가 있고, `network_subsystem.md`에 구역 전환 정책이 있지만, 이 둘이 동일 패킷인지 별개인지 연결이 안 되어 있다.

**기획서가 추가해야 할 것:**
- 구역 전환을 트리거하는 서버 패킷이 정확히 어떤 것인지 (`CheckpointReachedEvent`? 별도 `ZoneTransitionEvent`?)
- `DungeonEnterEvent`와 최초 구역 스폰의 관계 (DungeonEnterEvent 수신 후 스폰 패킷이 오는 것인지, 동시인지)
