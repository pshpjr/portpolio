# AGENTS.md
> `docs/proposal/`용 에이전트 진입점입니다.
> 이 디렉터리는 구현 규칙이 아니라 제품 기획 결정과 범위 정의를 담습니다.

---

## 목적

- 제품 방향과 우선순위를 빠르게 파악한다.
- 구현 전에 필요한 기능 범위와 용어를 확인한다.
- 세부 기획 문서를 필요한 만큼만 읽어 컨텍스트 사용량을 줄인다.

---

## 읽기 순서

1. [`main.md`](./main.md)에서 프로젝트 개요, 핵심 결정, 마일스톤만 확인한다.
2. [`exec-plans/active/INDEX.md`](./exec-plans/active/INDEX.md)에서 관련 활성 작업이 있는지 확인한다.
3. 수정 대상과 직접 관련된 하위 문서만 추가로 읽는다.
4. 구현 제약이 필요해지면 `server/` 같은 앱 문서로 내려간다.
5. 문서 수정 후에는 가능한 환경이면 문서 검증 스크립트를 실행한다.

---

## 문서 역할

- `main.md`: 허브 문서. 핵심 결정, 범위, 마일스톤, 하위 문서 지도만 유지한다.
- `exec-plans/`: 기획서 구조 개편과 진행 중 작업 문서를 관리한다.
- `common/`: 계정, 아이템, 스킬처럼 여러 영역이 공유하는 규칙을 정의한다.
- `ingame/`: 전투, 보스, AI, 던전 진행처럼 플레이 중 규칙을 정의한다.
- `outgame/`: 파티, 채팅, 경제, 입장 흐름처럼 허브/로비 규칙을 정의한다.
- `server/`: 구현이 아니라 서버 관점의 기획 의도와 기술 목표를 정의한다.

---

## 하위 문서 정리 원칙

- 한 문서는 한 주제만 다룬다.
- 첫 화면에서 알아야 할 내용만 위쪽에 둔다.
- 구현 세부보다 기획 결정, 플레이 규칙, 제약을 우선 적는다.
- 다른 문서와 중복되는 설명은 요약만 두고 원문 문서로 링크한다.
- 서버 구현 규칙, DB 스키마 확정, 패킷 정의처럼 구현 문서는 앱 디렉터리로 넘긴다.

---

## 권장 섹션

필요한 문서만 아래 순서를 따른다.

1. 목적 또는 한 줄 요약
2. 핵심 규칙
3. 주요 흐름 또는 상태 전이
4. 제약과 예외
5. 관련 문서
